**文档类型：** 决策支撑（As-Is / To-Be 对照）

**范围：** FW 主链词库 recall、领域判定、Lexicon V2 CPU Intent  
**读者：** 产品 / 架构 / 决策部门

**详细审计（含 Session 意图 SSOT 目标契约）：** [词库架构_Session意图与行业词库_只读审计报告_2026_05_30.md](./词库架构_Session意图与行业词库_只读审计报告_2026_05_30.md)

## 1. 执行摘要

| 维度 | 现状（As-Is） | 诉求（To-Be） |
|------|---------------|---------------|
| 存储形态 | 单一 `lexicon.sqlite`，启动时 **全表读入内存** | 基础词表 + 专业词表 **分表**，拼音键（如 `zhong\|bei`）为主键 |
| 领域判定 | `domain_anchor`（Detector 信号）+ 可选 CPU LLM Intent（profile 加分） | Session 主题关键词 → **行业表** 查表定域 |
| Recall 路径 | 一次拼音 bucket 查找 → domain 硬过滤 → profile 软加分 | 定域后 **分别查基础表 + 专业表** → 合并候选 |
| 规模扩展 | 7 万级可接受；百万级 **内存隐患真实存在** | 按需加载 / SQL 索引查表，控制内存上界 |
| CPU LLM Intent | **代码完整存在**，默认 `lexiconV2.enabled=false`，未随「代码丢失」删除 | 需决策：保留 LLM 定域 vs 改为行业关键词表 |

**核心结论：** 当前实现把 SQLite 当作 **bundle 载体 + 启动批量导入**，搜索靠 **内存 Map（拼音倒排）**；`domains` 字段是 **标签过滤/加权**，不是分库键。诉求中的「分表 + 行业表定域 + 双路 recall」是 **架构级变更**，需单独立项，且与 FW 主链冻结合约存在接口层改造面。

---

## 2. 背景与决策问题

### 2.1 业务诉求

1. 词库将持续扩张，行业（专业词表）数量不断增加。
2. **全量加载进内存** 在规模上升后成为隐患。
3. 希望词表以 **拼音化键**（如 `zhong bei` / `zhong|bei`）为主键，**基础词表**与**专业词表**分离存储。
4. 识别任务先用 Session 主题关键词在 **行业表** 定域，再分别从基础/专业表取候选，合并多选项。

### 2.2 需决策的问题

| # | 问题 |
|---|------|
| D1 | 是否批准「分表拼音主键 + 行业表定域 + 双路 recall」作为下一代词库 runtime 目标架构？ |
| D2 | 迁移策略：Big Bang 替换 vs 与现网并行（feature flag / 双读对比）？ |
| D3 | 领域判定：行业关键词表 vs 保留/增强 CPU LLM Intent vs 混合？ |
| D4 | 对 **FW 主链冻结**（`ASR → FW_SPAN_DETECTOR → AGGREGATION → …`）的例外范围？ |

## 3. 现状架构（As-Is）

### 3.1 数据与构建链路

```text
canonical_term JSONL（可分 base / idiom / domain_patch 等 seed 层）
  → lexicon:validate
  → lexicon:build（migrate + 写入 SQLite）
  → bundle: manifest.json + lexicon.sqlite + checksum.txt
  → 部署至 node_runtime/lexicon/current
```

- **Seed 分层** 仅存在于 **构建阶段**（工程维护便利），merge 后进入 **同一张** `lexicon_terms` 表。
- SQLite schema（build 时）：

```sql
CREATE TABLE lexicon_terms (
  id TEXT PRIMARY KEY,
  word TEXT NOT NULL,
  pinyin TEXT NOT NULL,
  domains TEXT,          -- JSON 数组标签，如 ["restaurant"]
  prior_score REAL,
  repair_target INTEGER,
  enabled INTEGER,
  ...
);
CREATE INDEX idx_lexicon_terms_word ON lexicon_terms(word);
-- 注意：无 (pinyin_key, domain) 复合索引；无分表
```

- `manifest.json` 中的 `pinyinIndexCount` 是 **构建期统计**，不是 SQLite 内倒排表；runtime 在内存重建索引。

**关键文件：**

| 环节 | 路径 |
|------|------|
| Build | `scripts/lexicon/build-lexicon-bundle.mjs`、`lib/build-bundle.mjs` |
| Runtime load | `main/src/lexicon/lexicon-runtime.ts` |
| FW recall | `main/src/lexicon/local-span-recall.ts` → `pinyin-topk-lookup.ts` |
| Domain 硬过滤 | `main/src/lexicon/domain-filter.ts` |
| Domain 软加分 | `main/src/lexicon/domain-boost-calculator.ts` |

### 3.2 Runtime 加载模型（内存）

```text
进程启动 / 首次 FW job
  → LexiconRuntime.load()
  → SELECT * FROM lexicon_terms          ← 全表一次（非 per-span）
  → buildHotwordPinyinIndex()            ← Map<pinyinKey, HotwordEntry[]>
  → buildExactWordIndex / buildAliasIndexes
  → 之后 recall 不再发 SQL
```

- **Recall 热路径不是全库遍历**：对每个 span 做 `getPinyinBucket("zhong|bei")`，复杂度 O(1) 查 Map + bucket 内少量词条打分。
- **隐患在启动/换 bundle**：词量 × 字段宽度 → 常驻内存线性增长；alias 索引同样随词条膨胀。

### 3.3 FW 主链上的 Recall 流程

```text
raw ASR text
  → detectSuspiciousSpansV1（含 domain_anchor_nearby 等检测信号，不查词库候选）
  → 对每个 span:
       textToSyllables → syllablesKey
       lookupTopKByPinyin → getPinyinBucket(key)   // 统一 bucket，含所有 domain 标签词条
       filter: minPrior
       filter: matchEnabledDomain(domains, enabledDomains)  // general 硬拒绝
       score: prior + phonetic + domainBoost(profile) + ...
       KenLM weak_veto → applySpanReplacements
  → ctx.segmentForJobResult（业务 SSOT）
```

**`enabledDomains`（FW 配置，默认 travel/transport/restaurant/tech_ai）** 是 recall **硬白名单**；与词条 `domains` 标签求交。

**`ActiveLexiconProfileSnapshot`（Lexicon V2 profile）** 在同一候选池内做 **domainBoost 加权**（primary +0.12，secondary +0.06 量级），不切换物理词表。

### 3.4 领域判定：现状三条线（均非「行业表查拼音」）

| 机制 | 阶段 | 作用 | 是否 recall 查词 |
|------|------|------|------------------|
| `domain_anchor.json` | FW Detector | span 附近锚词 → `domain_anchor_nearby` 风险信号 | 否 |
| `enabledDomains` | FW Recall | 词条 domain 标签硬过滤 | 是（过滤 bucket 内结果） |
| Lexicon V2 CPU LLM Intent | Session finalize 后异步 | 推断 primaryDomain → 更新 profile → 下 turn domainBoost | 否（不查词库） |
| `profile-registry.json` | 构建/校验/LLM 白名单 | 域 ID 注册 | 否 |

**重要：** 当前 **没有**「行业关键词表 + 拼音主键 → 定域 → 再查基础/专业表」的实现。

### 3.5 规模与实测参考

| 指标 | 参考值 |
|------|--------|
| P1.3 v2 combined seed | ~73,089 行 |
| 当前 deploy 阶梯（文档） | ~4,962 词（5k bundle） |
| FW dialog_200 批测（历史） | pipeline p50 ~2.17s；recall 非瓶颈 |
| 内存 | 未做 formal cap；73k 级在 Electron 节点可接受，**百万级无架构保障** |

### 3.6 现状优势

1. **热路径延迟稳定**：span recall 零 SQL，Map O(1)。
2. **同音跨域竞争正确**：`钟贝 → zhong|bei` bucket 内 `中杯/钟北/…` 同池比较，KenLM + domainBoost 决胜负；分 domain 早筛可能漏候选或需多路 union。
3. **Bundle 可版本化**：checksum + manifest，部署简单。
4. **与 FW 冻结合约已对齐**：`local-span-recall` + `domain-filter` 已批测 gate。

### 3.7 现状隐患（与诉求一致）

1. **启动全量 load**：词库越大，内存与启动时间越长，无按需卸载。
2. **SQLite 关系能力未用于 runtime 查询**：仅 `word` 索引，无拼音/domain 复合索引；与「用关系库做检索」的预期不符。
3. **domain 是标签不是分库**：专业词表膨胀仍进入同一 bucket，过滤发生在 recall 之后。
4. **多行业扩展**：每增行业需扩 `enabledDomains`、anchor、profile-registry，缺统一的「行业表」数据面。

## 4. 目标架构（To-Be，据业务诉求整理）

### 4.1 逻辑数据模型（提议）

```text
industry_lexicon          -- 行业/领域判定表
  PK: pinyin_key          -- 如 "zhong|bei" 或 session 主题词的拼音键
  fields: domain_id, anchor_terms, weight, ...

base_lexicon              -- 基础词表
  PK: pinyin_key
  SK/row: word, prior, repair_target, ...

domain_lexicon_{domain}   -- 各专业词表（或 domain_lexicon 单表 + domain 列 + 复合索引）
  PK: (domain_id, pinyin_key)
  fields: word, prior, repair_target, aliases, ...
```

### 4.2 Runtime 流程（提议）

```text
Session 开始 / turn 更新
  → 提取 session 主题关键词（来源待定：LLM summary / 热词 / 用户设定）
  → 拼音化 → 查 industry_lexicon → 得到 activeDomain(s)

对每个 FW span:
  → pinyin_key = syllablesKey(span)
  → candidates_base  = lookup(base_lexicon, pinyin_key)
  → candidates_domain = lookup(domain_lexicon, pinyin_key, activeDomain)
  → merge + dedupe + score + KenLM → pick
```

### 4.3 与现状的本质差异

| 点 | 现状 | 目标 |
|----|------|------|
| 主键 | 词条 `id`；拼音键仅内存 Map | 拼音键为 **表级主键/索引键** |
| 存储 | 单表 + 全量内存索引 | 分表或分区 + **按需 SQL / 分页 load** |
| 定域 | anchor 信号 + 可选 LLM profile | **行业表** 关键词查表 |
| Recall | 单 bucket 一次 | **至少两次 lookup**（base + domain）再 merge |
| 内存 | 全库常驻 | 可设 **上界**（LRU bucket cache、按 domain 挂载） |

### 4.4 目标方案潜在收益

1. **内存可控**：仅加载当前 session 涉及 domain 的专业子集 + 基础表索引/metadata。
2. **行业扩展线性**：新增 `domain_lexicon_medical` 不必重导全库到单一 mega-index。
3. **运维语义清晰**：基础 vs 专业职责分离，patch 不再 fear alias 与 base canonical 冲突（若分表隔离策略明确）。
4. **符合关系库初衷**：`(pinyin_key)` / `(domain_id, pinyin_key)` B-tree 索引，million 级可接受。

### 4.5 目标方案风险与挑战

| 风险 | 说明 |
|------|------|
| **同音跨表竞争** | `钟贝` 可能在 base 与 restaurant 均有候选；merge 规则、KenLM 输入句需重新定义 |
| **定域错误传播** | 行业表误判 → 专业表查错域 → recall 空或偏；需 fallback（base-only / 多域 union） |
| **延迟** | 双路 SQL + merge vs 现网单 Map；需 benchmark 预算 |
| **主链改造面** | `lexicon-runtime`、`local-span-recall`、`build-bundle`、FW orchestrator、gate 测试全链 |
| **与冻结冲突** | FW 主链步骤顺序不变，但 **recall 内核** 属实质变更，需冻结例外审批 |
| **Session 主题词来源** | 诉求依赖「主题关键词」— 现网无统一 SSOT，可能仍需 LLM 或规则抽取 |

## 5. 对照总表（决策用）

| 维度 | 现状 | 目标 | 改造量 |
|------|------|------|--------|
| Bundle 格式 | 单 sqlite + manifest | 多表 sqlite 或分文件 bundle | 高 |
| Build 管线 | validate → migrate → 单表 insert | 分表 build + 行业表生成 | 高 |
| Runtime load | 全表 → 内存 Map | 索引/metadata 常驻 + bucket 按需 | 高 |
| 定域 | anchor + LLM profile（可选） | 行业表 +（可选保留 LLM） | 中–高 |
| FW recall API | `recallSpanTopK(runtime, …)` | 新 API 或 runtime 内部分路 | 中 |
| 测试 | dialog_200 gate、freeze-contract | 全量回归 + 内存/规模压测 | 高 |
| 向后兼容 | 现网 v3 5k / P1.3 v2 seed | 需迁移工具与双轨期 | 中 |

## 6. 迁移路径建议（供决策选型）

### 方案 A：渐进式（推荐作为默认决策选项）

1. **Phase 0（不改 recall）**：build 时按 base/domain 分表写入同一 sqlite，runtime 仍全量 load — 仅验证数据模型与 validate 规则。
2. **Phase 1**：runtime 改为 **SQL 按 pinyin_key 查 base + domain 表**，内存只缓存 hot bucket（LRU）。
3. **Phase 2**：引入 **industry_lexicon** 定域；与 `domain_anchor` / LLM profile 做 shadow 对比。
4. **Phase 3**：切换 FW recall 到双路 merge；下线全量 `buildHotwordPinyinIndex`。

**优点：** 可分批 gate，回滚容易。  
**缺点：** 周期 long，需维护双实现一段时间。

### 方案 B：Big Bang

一次性替换 bundle 格式 + runtime + recall。

**优点：** 架构干净。  
**缺点：** 回归风险高，与 FW 冻结冲突大。

### 方案 C：维持现状 + 规模缓解（最小改动）

- 按 domain 拆 **多个 bundle**，session 绑定 profile 后只 load 对应 bundle + general base。
- 仍用内存 Map，但 **上限 = 单 bundle 词量**。

**优点：** 改动小于分表 SQL。  
**缺点：** 同音跨 bundle 仍需 union；多 bundle 运维复杂；未完全满足「拼音主键分表」诉求。

## 7. FW 主链与冻结合约影响

**冻结步骤（不可变顺序）：**

```text
ASR → FW_SPAN_DETECTOR → AGGREGATION → DEDUP → TRANSLATION
```

**可讨论变更面（需书面例外）：**

| 组件 | 是否触及冻结步骤 | 说明 |
|------|------------------|------|
| `lexicon-runtime.ts` | 否（FW 下游实现） | recall 数据源变更 |
| `local-span-recall.ts` | 否 | 双路 merge 逻辑 |
| `fw-detector-orchestrator.ts` | 否 | 可能增加定域前置调用 |
| `domain-filter.ts` | 否 | 规则可能简化或下沉到 SQL WHERE |
| KenLM gate / apply | **是（行为契约）** | pick 输入候选集变化 → 需 dialog_200 重批 |

**建议：** 若批准 To-Be，单独开 **「词库 Runtime V2」** 变更单，与 FW 主链冻结解耦审批；recall 输出契约（`segmentForJobResult` SSOT）保持不变。

## 8. CPU LLM 意图推理服务存续状态（专项答复）

### 8.1 结论：**未丢失，代码与 Python 服务均完整存在**

此前「代码丢失」**未波及** Lexicon V2 Intent 栈。仓库内可定位的完整链路：

| 层级 | 路径 | 状态 |
|------|------|------|
| Node 调度 | `main/src/lexicon-v2/intent-job-scheduler.ts` | ✅ |
| Worker | `main/src/lexicon-v2/cpu-intent-llm-worker.ts` | ✅ |
| HTTP 客户端 | `main/src/lexicon-v2/cpu-llm-model-runner.ts` | ✅ |
| Prompt / 解析 | `lexicon-intent-prompt-builder.ts`、`lexicon-profile-decision-parser.ts` | ✅ |
| Session 挂接 | `session-runtime/session-finalize.ts` → `enqueueIntentJob` | ✅ |
| 健康检查 / 恢复 | `intent-health-check.ts`、`intent-recovery.ts`、`intent-warmup.ts` | ✅ |
| Python 服务 | `services/lexicon_intent_cpu/service.py`（FastAPI，端口 **5018**） | ✅ |
| 服务注册 | `services/lexicon_intent_cpu/service.json`（id: `lexicon-intent-cpu`） | ✅ |
| 进程管理 | `service-layer/ServiceProcessRunner.ts`（含 model 路径解析） | ✅ |
| 预加载列表 | `ServiceProcessRunner-internal.ts` → `MODEL_PRELOAD_SERVICES` 含 `lexicon-intent-cpu` | ✅ |
| 单测 / smoke | `cpu-intent-llm-worker.test.ts`、`lexicon-intent-integration.smoke.test.ts` | ✅ |
| 模型目录说明 | `electron-node/models/lexicon-intent/README.md` | ✅ |

### 8.2 默认配置：**功能关闭，非代码缺失**

`node-config-defaults.ts`：

```json
"lexiconV2": {
  "enabled": false,
  "intentEnabled": true,
  "intentMode": "cpu_llm",
  "cpuWorker": {
    "serviceUrl": "http://127.0.0.1:5018",
    "modelPath": "models/lexicon-intent/qwen2.5-3b-instruct-q4_k_m.gguf"
  }
}
```

| 开关 | 含义 |
|------|------|
| `lexiconV2.enabled = false` | **默认不调度** Intent job；FW 使用 `defaultGeneralProfile()` |
| `intentEnabled = true` | 仅在 `enabled=true` 时生效 |
| GGUF 模型 | gitignore，需本地下载至 `electron-node/models/lexicon-intent/` |

### 8.3 运行时行为（enabled=true 时）

```text
session finalize
  → shouldScheduleIntentJob（bootstrap / 间隔 / no_topk_surge 等触发）
  → enqueueIntentJob（单 worker，latest-only 队列）
  → POST http://127.0.0.1:5018/intent
  → parseLexiconProfileDecision → stagePendingProfile
  → 下一 turn bindProfileSnapshotToContext → FW domainBoost
```

**与 FW recall 的关系：** LLM Intent **不查词库**、**不替代** `enabledDomains` 硬过滤；仅影响 **同 bucket 内排序加分**。

### 8.4 启用检查清单

1. 配置 `features.lexiconV2.enabled: true`
2. 放置 GGUF（如 `qwen2.5-3b-instruct-q4_k_m.gguf`）
3. 确认 `lexicon-intent-cpu` 服务运行，`GET /health` → `model_loaded: true`
4. 观察 `session.extra` / intent diagnostics 中 `intentLastOutcome`

## 9. 决策建议（起草）

| 选项 | 建议 |
|------|------|
| **是否认可内存隐患** | 认可；73k 可撑，**10万+ / 多行业并行** 需架构演进 |
| **是否立即改 To-Be** | 建议 **立项 Phase 0/1**，不与 P1.3 seed 导入阻塞 |
| **定域机制** | 行业表与 CPU LLM **可并存**：行业表做 fast path，LLM 做 slow path 校正；避免二选一硬切 |
| **CPU LLM** | **保留**；默认关闭是产品策略，非代码缺失 |
| **FW 冻结** | recall 内核变更走 **例外变更单** + dialog_200 全量回归 |
