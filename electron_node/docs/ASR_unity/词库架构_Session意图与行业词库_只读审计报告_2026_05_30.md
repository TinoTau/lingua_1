# 词库架构 · Session 意图 · 行业词库 — 只读审计报告

**版本：** V1.0（只读，本报告不实施代码变更）  
**日期：** 2026-05-30  
**审计范围：** 词库 Runtime、FW Recall、Session 对象、Lexicon V2 CPU LLM Intent  
**关联文档：** [词库分表拼音主键架构_现状与目标_决策报告_2026_05_30.md](./词库分表拼音主键架构_现状与目标_决策报告_2026_05_30.md)  
**主链状态：** FW 主链步骤顺序已冻结；本报告涉及的 recall / session 字段 / 词库存储属 **待立项改造面**

---

## 执行摘要

| 审计项 | 现状结论 | 目标需求 |
|--------|----------|----------|
| 词库存储 | 单一 `lexicon.sqlite`，启动 **全表读入内存**，拼音 Map 检索 | 基础词表 + 专业词表 **分表**，拼音键（`zhong\|bei`）为主键/索引键，**按需查询** |
| 行业判定 | `domain_anchor`（Detector 信号）、`enabledDomains`（硬过滤）、profile **软加分** | **Session 意图字段** → 行业表/专业词表路由 |
| CPU LLM Intent | **代码完整**，默认 `lexiconV2.enabled=false`；写入 `lexiconIntentSummary` + `activeLexiconProfile` | **必须**将用户意图结构化写入 Session，作为行业词库判定的 **SSOT** |
| LLM → 行业词库 | **未接通**：profile 仅用于 `domainBoost`，不选择专业词表 | LLM 输出 `primaryDomain` + `topicKeywords` → Session → 行业/专业 recall |
| 内存隐患 | 7 万级可接受；无规模上界 | 分表 + 按需 load / SQL 索引，控制常驻内存 |

**核心缺口：** CPU LLM 已能把 **域 ID** 和 **文本 summary** 写入 Session，但 **没有结构化「用户意图/主题关键词」字段**，也 **没有** 任何组件用 Session 意图去查行业词表。目标架构要求补齐 **Session 意图 SSOT → 行业词库判定** 这条链。

## 二、现状：词库 Runtime

### 2.1 构建与 Bundle

```text
canonical_term JSONL（构建期可分层：base / idiom / domain_patch）
  → npm run lexicon:validate
  → npm run lexicon:build
  → node_runtime/lexicon/current/
       manifest.json
       lexicon.sqlite
       checksum.txt
```

- Seed 分层 **仅存在于构建**；merge 后写入 **同一张** `lexicon_terms` 表。  
- SQLite 仅 `idx_lexicon_terms_word`；**无** `(pinyin_key, domain)` 索引，**无** base/domain 分表。

**SSOT 文件：**

| 环节 | 路径 |
|------|------|
| Build | `scripts/lexicon/lib/build-bundle.mjs` |
| Runtime | `main/src/lexicon/lexicon-runtime.ts` |
| FW recall | `main/src/lexicon/local-span-recall.ts` |

### 2.2 启动加载（全量内存）

```text
LexiconRuntime.load()
  → SELECT id, word, pinyin, domains, prior_score, ... FROM lexicon_terms   // 全表
  → buildHotwordPinyinIndex()     // Map<"zhong|bei", HotwordEntry[]>
  → buildExactWordIndex()
  → buildAliasIndexes()
  → 之后 recall 不再发 SQL
```

| 阶段 | 复杂度 | 说明 |
|------|--------|------|
| 启动 load | O(N) N=词条数 | **隐患点**：词库扩张 → 内存线性涨 |
| 单次 span recall | O(1) + O(bucket) | **不是**全库遍历；bucket 通常很小 |

### 2.3 FW Recall 热路径

```text
detectSuspiciousSpansV1
  → recallSpanTopK(span, profile, enabledDomains)
      → lookupTopKByPinyin → getPinyinBucket(pinyinKey)
      → filter minPrior
      → filter matchEnabledDomain(domains, enabledDomains)   // general 硬拒绝
      → score: prior + phonetic + computeDomainBoost(profile)
      → KenLM → apply
```

- `domains: ["restaurant"]` 是词条 **标签**，不是分表键。  
- **所有 domain 词条在同一拼音 bucket**，过滤发生在 bucket 内。

模型需本地下载：`electron-node/models/lexicon-intent/*.gguf`（gitignore）。

### 4.2 LLM 推理触发策略

`shouldScheduleIntentJob`（`intent-job-scheduler.ts`）：

| 触发器 | 条件 |
|--------|------|
| `bootstrap` | 前 3 个 finalized turn |
| `interval_refresh` | 每 20 turn |
| `time_refresh` | 距上次 intent ≥ 3 分钟 |
| `no_topk_surge` | 连续 3 turn 无 TopK 且比例 > 50% |

### 4.3 LLM 输出 schema（当前）

解析器：`lexicon-profile-decision-parser.ts` → `LexiconProfileDecision`

| 字段 | 类型 | 说明 |
|------|------|------|
| `summary` | string | 对话意图摘要（≤300 字） |
| `primaryDomain` | string | 须在 `profile-registry.json` 且 `allowLLMSelect` |
| `secondaryDomains` | string[] | 最多 2 个 |
| `confidence` | number | 0–1 |
| `shouldSwitch` | boolean | 是否建议切域 |
| `reason` | string[] | 理由 |
| `effectiveFromTurn` | number | 生效 turn |

**当前 LLM 不输出：** `topicKeywords`、`scenarioTags`、行业锚词列表。

### 4.4 Session 写入链（当前）

```text
finalizeSessionTurn
  → scheduleIntentIfNeeded
  → enqueueIntentJob → inferLexiconProfileDecision → POST :5018/intent
  → parseLexiconProfileDecision
  → 写入 Session:
       lexiconIntentSummary = { summary, updatedAt }          // 仅摘要
       applyProfileDecision → pendingProfile / profileHistory
       lastIntentAtMs
       intentDiagnostics（可观测性）

下一 turn beginTurnForJob:
  → activatePendingProfileForTurn
  → bindProfileSnapshotToContext(ctx, snapshot)
  → FW: getProfileSnapshotFromContext(ctx) ?? defaultGeneralProfile()
       → computeDomainBoost（仅加分，不选表）
```

### 4.5 Session 意图相关字段一览（当前）

| Session 字段 | 类型 | 写入方 | 下游消费者 | 用于行业词库？ |
|--------------|------|--------|------------|----------------|
| `lexiconIntentSummary` | `{ summary, updatedAt }` | Intent job 成功 | 迁移 export、可观测 | **否** |
| `activeLexiconProfile` | `primaryDomain`, `secondaryDomains`, `boosts`, `confidence` | profile 激活 | FW `domainBoost` | **间接**（仅加分） |
| `pendingProfile` | 同上 | Intent 决策 staging | 下 turn 激活 | 同上 |
| `turnProfileSnapshot` | 同上 | `bindProfileSnapshotToContext` | 当前 job FW | 同上 |
| `profileHistory` | 切换事件 | Intent 应用 | 审计 | 否 |
| `intentDiagnostics` | 健康/ outcome | recordIntent* | `result.extra` | 否 |
| `lastIntentAtMs` | number | Intent job | 调度间隔 | 否 |

**审计结论：** Session 已有 **域 ID**（profile）和 **自然语言 summary**，但 **缺少结构化用户意图/主题关键词**，且 **无任何 recall 路径读取这些字段做行业词表路由**。

### 4.6 默认关闭的影响

`lexiconV2.enabled=false` 时：

- 不调度 Intent job  
- FW 始终 `defaultGeneralProfile()`（`primaryDomain: 'general'`）  
- `lexiconIntentSummary` 不更新  
- **表现如同「LLM 意图服务不存在」**，但代码仍在

## 六、目标需求（业务 + 技术）

### 6.1 词库结构目标

```text
base_lexicon
  PK / Index: pinyin_key   -- 如 "zhong|bei"
  字段: word, prior_score, repair_target, ...

domain_lexicon_{domain}   -- 或 domain_lexicon(domain_id, pinyin_key)
  PK / Index: (domain_id, pinyin_key)
  字段: word, prior_score, repair_target, aliases, ...

industry_routing_lexicon   -- 行业路由/锚词（可选独立表）
  Index: pinyin_key 或 keyword
  字段: domain_id, weight, ...
```

**Recall 目标流程：**

```text
1. 从 Session 读取意图 SSOT → 得到 activeDomain + topicKeywords（拼音化）
2. （可选）用 topicKeywords 查 industry_routing_lexicon 校验/修正 domain
3. 对 FW span: pinyin_key = syllablesKey(span)
4. candidates = lookup(base_lexicon, pinyin_key)
              ∪ lookup(domain_lexicon, pinyin_key, activeDomain)
5. merge → score → KenLM → pick
```

### 6.2 Session 意图 SSOT 目标（核心新增契约）

**原则：** CPU LLM 推理的用户意图 **必须** 写入 Session 专用字段，作为 **行业词库判定的唯一业务输入**（SSOT）。FW recall 与行业表查询 **不得** 绕过 Session 直读 LLM。

#### 6.2.1 建议新增/扩展类型

```typescript
/** Session 级用户意图 SSOT — 由 CPU LLM 写入，行业词库/recall 只读 */
export type LexiconSessionIntent = {
  /** 自然语言摘要（保留现有 summary 语义） */
  summary: string;
  /** 主题关键词（中文），用于行业表/路由表拼音化查询 */
  topicKeywords: string[];
  /** 预计算拼音键，与 syllablesKey 一致，如 "zhong|bei" */
  topicKeywordPinyinKeys: string[];
  /** 主行业域 ID，与 profile-registry 对齐 */
  primaryDomain: string;
  secondaryDomains: string[];
  confidence: number;
  updatedAt: number;
  effectiveFromTurn: number;
  source: 'cpu_llm' | 'manual' | 'fallback_anchor';
  /** LLM 原始 reason，便于审计 */
  reason: string[];
};

export type SessionObject = {
  // ...现有字段...
  /** 目标 SSOT：取代或扩展现有 lexiconIntentSummary */
  lexiconSessionIntent?: LexiconSessionIntent;
  activeLexiconProfile: ActiveLexiconProfileSnapshot;  // 与 intent 同步，供 turn 内固定
};
```

#### 6.2.2 字段职责划分（目标）

| 字段 | 职责 | 写入方 | 读取方 |
|------|------|--------|--------|
| `lexiconSessionIntent` | **用户意图 SSOT**（摘要 + 关键词 + 域） | CPU LLM Intent job | 行业表查询、专业词表路由、可观测 |
| `activeLexiconProfile` | **Turn 内固定** profile（含 boosts） | Intent 决策激活 | FW `domainBoost`、turn binding |
| `turnProfileSnapshot` | JobContext 绑定 | `beginTurnForJob` | FW orchestrator |
| `lexiconIntentSummary` | （ deprecated ） | — | 迁移期双写后可删 |

#### 6.2.3 CPU LLM 输出 schema 目标扩展

在现有 `LexiconProfileDecision` 上增加：

| 新字段 | 类型 | 用途 |
|--------|------|------|
| `topicKeywords` | `string[]` | 3–8 个中文主题词，如 `["咖啡","中杯","外卖"]` |
| `topicKeywordPinyinKeys` | `string[]` | Node 侧或由 LLM 给出；**建议 Node 用 `textToSyllables` 统一计算** |

Python prompt / parser（`intent_engine.py`、`lexicon-profile-decision-parser.ts`）需同步扩展。

#### 6.2.4 写入时机（目标）

```text
Intent job 成功
  → 解析 decision + topicKeywords
  → Node 计算 topicKeywordPinyinKeys
  → session.lexiconSessionIntent = { ... }     // SSOT 更新
  → sync activeLexiconProfile / pendingProfile   // 与 intent.primaryDomain 一致
  → session migration export 含 lexiconSessionIntent

beginTurnForJob
  → 若 lexiconSessionIntent.effectiveFromTurn <= turnNumber
  → bindProfileSnapshotToContext
  → ctx.lexiconSessionIntent = clone(session.lexiconSessionIntent)   // JobContext 只读副本

FW recall / 行业词表（目标）
  → activeDomain = ctx.lexiconSessionIntent?.primaryDomain ?? profile.primaryDomain
  → routingKeys = ctx.lexiconSessionIntent?.topicKeywordPinyinKeys ?? []
  → lookup industry + base + domain tables
```

#### 6.2.5 Fallback 策略（目标）

| 条件 | 行为 |
|------|------|
| `lexiconV2` 关闭或 LLM 失败 | `lexiconSessionIntent.source = 'fallback_anchor'`，用 `domain_anchor` + `enabledDomains` 推断 |
| `confidence < 0.75` | 不切换 domain；仍更新 `summary`/`topicKeywords` 供 observability |
| 无 `topicKeywords` | 仅用 `primaryDomain` 路由专业表；跳过 industry_routing 表 |

### 6.3 与 CPU LLM 的分工（目标）

```text
┌─────────────────────┐
│ CPU LLM Intent      │  输入: rollingContext (ASR 文本)
│ (lexicon_intent_cpu)│  输出: summary + topicKeywords + primaryDomain
└──────────┬──────────┘
           │ POST /intent → parse → 写入 Session
           ▼
┌─────────────────────┐
│ Session SSOT        │  lexiconSessionIntent（意图 + 域 + 关键词拼音键）
│ activeLexiconProfile│  turn 内 profile / boosts
└──────────┬──────────┘
           │ beginTurnForJob → JobContext
           ▼
┌─────────────────────┐
│ 行业词库判定        │  topicKeywordPinyinKeys → industry_routing_lexicon
│                     │  primaryDomain → domain_lexicon 路由
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ FW span recall      │  base_lexicon ∪ domain_lexicon(pinyin_key)
└─────────────────────┘
```

**要点：** LLM **不负责** pick 具体 repair 词条；只负责 **Session 级意图与行业域**。词条候选仍由拼音键 + KenLM 决定。
